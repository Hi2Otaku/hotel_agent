  <action>
    **Create multi-stage Dockerfiles for both frontends:**

    1. `frontend/Dockerfile` -- Guest frontend production build:
       ```dockerfile
       FROM node:20-alpine AS build
       WORKDIR /app
       COPY package*.json ./
       RUN npm ci
       COPY . .
       RUN npm run build

       FROM nginx:stable-alpine
       COPY --from=build /app/dist /usr/share/nginx/html
       COPY nginx.conf /etc/nginx/conf.d/default.conf
       EXPOSE 80
       ```

    2. `frontend/nginx.conf` -- SPA routing for guest frontend:
       ```nginx
       server {
           listen 80;
           root /usr/share/nginx/html;
           index index.html;

           location / {
               try_files $uri $uri/ /index.html;
           }

           location /assets/ {
               expires 1y;
               add_header Cache-Control "public, immutable";
           }
       }
       ```

    3. `frontend-staff/Dockerfile` -- identical structure to guest frontend Dockerfile (FROM node:20-alpine build stage, FROM nginx:stable-alpine serve stage, same COPY pattern).

    4. `frontend-staff/nginx.conf` -- SPA routing for staff frontend. Same as guest nginx.conf but with `location /staff/` handling. Since staff frontend is served at /staff/ path via the main Nginx reverse proxy, configure:
       ```nginx
       server {
           listen 80;
           root /usr/share/nginx/html;
           index index.html;

           location / {
               try_files $uri $uri/ /index.html;
           }

           location /assets/ {
               expires 1y;
               add_header Cache-Control "public, immutable";
           }
       }
       ```

    **Important:** Do NOT add a .dockerignore that excludes node_modules -- the COPY . . is after npm ci so node_modules from the build stage is correct. However, add `.dockerignore` files if they don't exist to exclude `node_modules/` and `dist/` from the COPY context for faster builds.
  </action>

    **1. Create `docker-compose.prod.yml`:**

    Production override that layers on top of docker-compose.yml:

    ```yaml
    services:
      auth:
        restart: always
        environment:
          FIRST_ADMIN_EMAIL: ${ADMIN_EMAIL:-admin@hotel.local}
          FIRST_ADMIN_PASSWORD: ${ADMIN_PASSWORD:-admin123}

      room:
        restart: always
        environment:
          SEED_ON_STARTUP: "true"

      booking:
        restart: always

      gateway:
        restart: always
        ports: []  # Remove host port mapping; Nginx handles routing

      frontend-guest:
        build:
          context: ./frontend
          dockerfile: Dockerfile
        restart: always

      frontend-staff:
        build:
          context: ./frontend-staff
          dockerfile: Dockerfile
        restart: always

      nginx:
        image: nginx:stable-alpine
        ports:
          - "80:80"
          - "443:443"
        volumes:
          - ./nginx/conf.d:/etc/nginx/conf.d:ro
          - letsencrypt_data:/etc/letsencrypt:ro
        depends_on:
          - gateway
          - frontend-guest
          - frontend-staff
        restart: always

    volumes:
      letsencrypt_data:
    ```

    Key decisions:
    - Gateway port mapping removed (empty ports list overrides dev mapping); Nginx proxies to gateway:8000 internally
    - All services get `restart: always` for production resilience
    - SSL volume uses named volume (letsencrypt_data) -- user runs certbot separately on EC2
    - SEED_ON_STARTUP stays "true" for demo purposes (portfolio piece needs data)
    - Admin credentials configurable via env vars with sensible defaults

    **2. Create `nginx/conf.d/default.conf`:**

    Reverse proxy configuration:

    ```nginx
    # HTTP -> HTTPS redirect (when SSL is configured)
    # Uncomment when Let's Encrypt is set up:
    # server {
    #     listen 80;
    #     server_name _;
    #     return 301 https://$host$request_uri;
    # }

    server {
        listen 80;
        # listen 443 ssl;  # Uncomment when SSL is configured
        server_name _;

        # SSL configuration (uncomment when Let's Encrypt is set up):
        # ssl_certificate /etc/letsencrypt/live/YOUR_DOMAIN/fullchain.pem;
        # ssl_certificate_key /etc/letsencrypt/live/YOUR_DOMAIN/privkey.pem;

        # API Gateway
        location /api/ {
            proxy_pass http://gateway:8000;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
            proxy_read_timeout 30s;
        }

        # Swagger docs passthrough
        location /docs {
            proxy_pass http://gateway:8000;
            proxy_set_header Host $host;
        }
        location /openapi.json {
            proxy_pass http://gateway:8000;
            proxy_set_header Host $host;
        }

        # Staff frontend (must come before guest frontend catch-all)
        location /staff/ {
            proxy_pass http://frontend-staff:80/;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }

        # Health check endpoint
        location /health {
            proxy_pass http://gateway:8000;
        }

        # Guest frontend (default catch-all)
        location / {
            proxy_pass http://frontend-guest:80;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }
    }
    ```

    Key routing:
    - `/api/*` -> gateway:8000 (backend API)
    - `/docs`, `/openapi.json` -> gateway:8000 (Swagger UI)
    - `/staff/*` -> frontend-staff:80 (staff SPA)
    - `/health` -> gateway:8000 (health check for CI/deploy verification)
    - `/*` -> frontend-guest:80 (guest SPA, catch-all)
    - SSL commented out by default -- user enables after running certbot on EC2
